# Hayulo Whitepaper

## Abstract

Hayulo is an experimental open-source programming language and toolchain for AI-assisted software creation. It is designed around a development loop in which humans express intent, LLMs generate and repair code, the compiler provides structured diagnostics, tests verify behavior, and humans review the result.

The project begins as a small prototype, but its larger goal is to make software creation more accessible, reliable, inspectable, and safe in a world where code is increasingly generated by AI systems.

## 1. Motivation

Software development is changing. LLMs can produce useful code from natural language descriptions, but the surrounding workflow remains fragile. Generated code often fails because of missing dependencies, ambiguous framework conventions, weak tests, runtime errors, unclear permissions, and diagnostics that were written only for human readers.

Existing languages are powerful, but they were not designed around AI generation and repair. Hayulo explores what a language might look like if that workflow were a first-class design constraint.

## 2. Core thesis

The central thesis of Hayulo is:

> Programming languages for the AI era should preserve human intent, produce readable code, provide machine-readable diagnostics, support built-in tests, expose risky effects, and enable safe repair loops.

This does not mean removing humans from software development. It means giving humans better leverage while keeping the process inspectable.

## 3. The Hayulo workflow

Hayulo targets this loop:

```text
Human describes intent
  -> LLM generates Hayulo code
  -> Hayulo compiler checks code
  -> Hayulo test runner verifies behavior
  -> structured diagnostics guide repair
  -> LLM or human applies patch
  -> human reviews and ships
```

This loop is the product. The language syntax, compiler, standard library, and package system should all serve it.

## 4. Design pillars

### 4.1 Intent preservation

Hayulo includes `intent` blocks as first-class metadata:

```hayulo
intent {
  purpose: "Manage invoices for freelancers."
  constraints: [
    "Never delete invoices permanently.",
    "All money values must include currency."
  ]
}
```

These blocks help future humans and models understand why code exists.

### 4.2 Predictable syntax

Hayulo favors a small, consistent syntax with braces for blocks, no semicolons by default, explicit public API types, and limited redundant patterns.

The goal is code that humans can read and LLMs can generate reliably.

### 4.3 Built-in tests

Tests are not an afterthought:

```hayulo
test "add works" {
  expect add(2, 3) == 5
}
```

A repair loop without tests is weak. Hayulo should make tests normal and easy.

### 4.4 Structured diagnostics

Hayulo diagnostics should be designed for both humans and machines:

```json
{
  "code": "type.mismatch",
  "message": "Expected Int but found Text.",
  "location": {
    "file": "src/main.hayulo",
    "line": 12,
    "column": 8
  },
  "suggestions": [
    {
      "kind": "convert_value",
      "message": "Use Int.parse(value)? if the text is numeric."
    }
  ]
}
```

Stable diagnostic codes can become an API for repair tools.

### 4.5 Static checking with practical inference

Hayulo should catch errors early while keeping code concise. Public APIs should be explicit. Local variables should be inferred.

Missing values should use `Option<T>`. Recoverable failures should use `Result<T, E>`.

### 4.6 Effects and permissions

Generated code should not silently gain dangerous capabilities. Hayulo should eventually track effects such as file writes, network calls, database writes, email sending, and money spending.

Project policy can then allow, deny, or require approval for effects.

### 4.7 Batteries-included app building

Hayulo should include official libraries for common tasks: JSON, files, CLI apps, HTTP, SQLite, validation, logging, configuration, tests, and eventually AI model calls.

A small useful app should not require a maze of unvetted dependencies.

## 5. Current prototype

The seed prototype includes:

- lexer
- parser
- AST
- tree-walking interpreter
- CLI
- examples
- tests
- JSON diagnostics

It supports only a small subset of the future language. This is intentional. The prototype is a base for iteration, not a finished implementation.

## 6. Why Hayulo may be useful

Hayulo can provide value if it makes the following easier:

- creating small useful tools
- repairing generated code
- preserving intent during modifications
- producing tests alongside code
- exposing risky effects
- teaching programming with AI assistance
- studying LLM coding reliability

## 7. Risks

The project faces real risks:

- new language adoption is hard
- ecosystems take years to build
- AI-generated code remains unreliable in many settings
- too many features too early can damage coherence
- safety claims can be overstated

Hayulo should address these risks through narrow early use cases, honest status labels, strong docs, tests, and evidence-driven design.

## 8. Roadmap summary

Hayulo should grow through staged releases:

1. seed prototype
2. useful scripting core
3. static checking preview
4. project system
5. formatter and repair protocol
6. app-building preview
7. effects and permissions
8. package ecosystem
9. public alpha
10. stable core

## 9. Conclusion

Hayulo is an experiment in programming language design for the AI-assisted era.

Its goal is not to eliminate programmers or hide code. Its goal is to make code generated with AI more structured, checkable, testable, repairable, and understandable.

If successful, Hayulo can help more people turn intent into software while keeping humans in control of meaning, risk, and review.
